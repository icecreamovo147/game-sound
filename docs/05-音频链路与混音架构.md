# 05-音频链路与混音架构

## 1. 音频链路目标

GameSound TUI 的核心音频目标是：

```text
真实麦克风输入 + 软件音效播放 → 实时混音 → 虚拟麦克风输出 → 语音软件
```

语音软件选择虚拟音频设备作为输入源后，队友可以同时听到用户的真实说话声和快捷键触发的音效。

## 2. 总体音频链路

```text
┌─────────────────┐
│ 真实麦克风设备   │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 麦克风采集模块   │
└────────┬────────┘
         │
         ▼
┌─────────────────┐        ┌─────────────────┐
│ 麦克风处理模块   │        │ 音效播放模块     │
│ 音量/静音/增益   │        │ 解码/队列/音量   │
└────────┬────────┘        └────────┬────────┘
         │                          │
         └──────────┬───────────────┘
                    ▼
           ┌─────────────────┐
           │ 实时混音模块     │
           └────────┬────────┘
                    │
          ┌─────────┴─────────┐
          ▼                   ▼
┌─────────────────┐   ┌─────────────────┐
│ 虚拟麦克风输出   │   │ 本地监听输出     │
└────────┬────────┘   └─────────────────┘
         ▼
┌─────────────────┐
│ Discord/QQ/游戏语音 │
└─────────────────┘
```

## 3. 麦克风采集模块

### 3.1 职责

- 枚举输入设备；
- 打开用户选择的麦克风；
- 读取实时音频帧；
- 转换采样格式；
- 处理设备断开；
- 提供麦克风电平数据。

### 3.2 输入格式

不同设备可能提供不同格式：

- 采样率：44100Hz、48000Hz、96000Hz；
- 声道：单声道、双声道；
- 样本格式：i16、u16、f32。

内部建议统一为：

```text
SampleFormat: f32
SampleRate: 48000Hz
Channels: 2
```

## 4. 音效播放模块

### 4.1 职责

- 加载音频文件；
- 解码音频帧；
- 重采样；
- 声道转换；
- 维护播放进度；
- 控制暂停、停止、循环；
- 将音效帧推送到混音器。

### 4.2 音效实例

一个音效触发后应创建播放实例：

```text
SoundInstance
- instance_id
- sound_id
- cursor
- volume
- loop
- status
- started_at
```

这样可以支持同一个音效被多次叠加播放。

## 5. 混音模块

### 5.1 基础混音公式

```text
mixed_sample = mic_sample * mic_volume + sum(sfx_sample_i * sfx_volume_i)
```

### 5.2 限幅

混音后需要限制在 `-1.0 ~ 1.0` 之间：

```text
output_sample = clamp(mixed_sample, -1.0, 1.0)
```

完整版本建议加入软限幅器，减少硬削波造成的失真。

### 5.3 多路音效

混音器应维护活跃音效列表：

- 播放中的音效；
- 循环音效；
- 队列中的音效；
- 即将淡出的音效。

每个音效帧与麦克风帧对齐后进行混音。

## 6. Ducking 设计

当音效播放时，自动压低麦克风音量。

流程：

```text
检测到音效开始播放
→ 在 attack_ms 内将麦克风音量降低到 ducking_ratio
→ 音效播放期间保持降低
→ 音效结束后等待 release_delay_ms
→ 在 release_ms 内恢复麦克风音量
```

配置示例：

```toml
[mixer.ducking]
enabled = true
ratio = 0.4
attack_ms = 50
release_ms = 300
release_delay_ms = 200
```

## 7. 虚拟麦克风输出

### 7.1 输出原理

程序并不能直接写入物理麦克风。应将混音结果写入一个虚拟音频输出设备，再由语音软件将对应虚拟设备作为输入源。

### 7.2 平台方案

| 平台 | 推荐方案 |
|---|---|
| Windows | VB-CABLE / Voicemeeter |
| macOS | BlackHole / Loopback |
| Linux | PulseAudio / PipeWire 虚拟设备 |

### 7.3 输出设备命名提示

系统可通过设备名称识别常见虚拟设备：

- VB-CABLE；
- CABLE Input；
- BlackHole；
- Loopback；
- Voicemeeter；
- PulseAudio Virtual Sink；
- PipeWire Virtual Device。

## 8. 本地监听输出

本地监听用于让用户自己听到音效或混音。

监听模式：

1. 仅监听音效；
2. 监听完整混音；
3. 监听麦克风和音效分离音量；
4. 关闭监听。

推荐默认监听模式：仅监听音效，避免用户听到自己的延迟回声。

## 9. 延迟控制

影响延迟的因素：

- 音频设备缓冲区大小；
- 解码延迟；
- 重采样延迟；
- 混音线程调度；
- 虚拟设备延迟；
- 语音软件采集延迟。

建议：

- 音频输出缓冲区优先使用低延迟配置；
- 音效可预解码或缓存前几帧；
- TUI 渲染线程不得阻塞音频线程；
- 所有控制事件通过无阻塞通道传递。

## 10. 线程模型

```text
主线程：TUI 渲染与用户输入
音频输入线程：采集麦克风
音效解码线程：加载与解码音频
混音输出线程：实时混音并写入输出设备
快捷键线程：监听全局快捷键
事件线程：统一处理状态变更
```

## 11. 错误处理

| 错误 | 处理方式 |
|---|---|
| 麦克风打开失败 | 提示重新选择设备 |
| 输出设备打开失败 | 停止输出并提示配置 |
| 音频解码失败 | 标记音频不可播放 |
| 采样率不匹配 | 尝试重采样 |
| 混音溢出 | 限幅处理并记录告警 |
| 设备断开 | 自动切换默认设备或暂停音频引擎 |
