# 06-技术架构设计

## 1. 技术栈

推荐技术栈：

```text
Rust
Ratatui
Crossterm
cpal
Symphonia
rusqlite
serde + toml
tracing
global-hotkey / rdev
tokio 或 crossbeam-channel
```

## 2. 架构原则

- TUI 与音频核心解耦；
- 音频线程不依赖 UI 状态；
- 所有控制通过事件和命令传递；
- 配置与数据库独立；
- 方便后续复用 core 做 GUI；
- 音频实时路径尽量无锁或低锁。

## 3. 模块划分

```text
gamesound-core
  audio/      音频输入、输出、解码、混音
  device/     设备枚举和设备选择
  hotkey/     全局快捷键注册与事件
  sound/      音效实体、播放实例、音频库
  mixer/      混音、限幅、ducking、电平统计
  runtime/    运行时控制器和事件总线
  config/     配置读写

gamesound-storage
  db/         SQLite 访问
  migration/ 迁移脚本
  backup/    配置备份

gamesound-tui
  app/        TUI 状态
  ui/         页面渲染
  widgets/    组件
  modal/      弹窗
  keymap/     键盘映射
  event/      终端事件循环
```

## 8. 音频核心设计

音频核心应提供稳定 API：

```rust
pub trait AudioEngine {
    fn start(&mut self) -> Result<()>;
    fn stop(&mut self) -> Result<()>;
    fn play_sound(&self, sound_id: i64) -> Result<()>;
    fn stop_all(&self) -> Result<()>;
    fn set_mic_device(&mut self, device_id: &str) -> Result<()>;
    fn set_output_device(&mut self, device_id: &str) -> Result<()>;
    fn set_monitor_device(&mut self, device_id: &str) -> Result<()>;
    fn set_volume(&mut self, target: VolumeTarget, value: f32) -> Result<()>;
}
```

## 9. 数据流

### 9.1 播放音效

```text
用户按 Enter 或全局快捷键
→ TUI/Hotkey 发出 PlaySound 命令
→ Runtime Controller 查询音频路径
→ Audio Engine 创建播放实例
→ Decoder 解码音频帧
→ Mixer 将音效帧与麦克风帧混合
→ Output Stream 写入虚拟设备
→ RuntimeEvent 更新状态
```

### 9.2 切换设备

```text
用户选择设备
→ RuntimeCommand::SetOutputDevice
→ AudioEngine 停止旧输出流
→ 打开新输出流
→ 更新配置
→ 通知 TUI
```

## 10. 并发策略

建议使用通道通信：

- TUI → Runtime：command_tx；
- Runtime → TUI：event_rx；
- Hotkey → Runtime：hotkey_tx；
- Audio → Runtime：audio_event_tx。

音频回调中不得执行阻塞 I/O、数据库访问或复杂日志写入。

## 11. 日志体系

日志输出到文件而不是 TUI 标准输出，避免破坏界面。

日志目录：

```text
~/.gamesound/logs/
```

日志文件：

```text
gamesound-YYYY-MM-DD.log
```

## 12. 后续 GUI 复用

后续 Tauri GUI 可以直接复用：

- gamesound-core；
- gamesound-storage；
- RuntimeCommand；
- RuntimeEvent；
- 数据模型。

GUI 仅替换 TUI 渲染层。
