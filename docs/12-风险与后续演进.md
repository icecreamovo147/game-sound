# 12-风险与后续演进

## 1. 技术风险

### 1.1 虚拟麦克风依赖风险

第一阶段依赖第三方虚拟音频设备。不同用户环境中设备名称、通道、延迟和权限可能不同。

应对：

- 提供设备检测；
- 标记疑似虚拟设备；
- 提供配置向导；
- 提供 doctor 诊断命令；
- 编写平台配置指南。

### 1.2 音频延迟风险

混音链路涉及麦克风采集、音效解码、重采样、虚拟设备和语音软件采集，可能产生延迟。

应对：

- 低延迟缓冲配置；
- 预加载常用音效；
- 音频线程独立运行；
- 避免 UI 阻塞音频；
- 提供延迟配置选项。

### 1.3 跨平台快捷键风险

全局快捷键在 Windows、macOS、Linux 上权限和实现差异较大。

应对：

- 抽象 hotkey 模块；
- 针对平台做适配；
- macOS 提供权限引导；
- 注册失败时提示原因。

### 1.4 TUI 兼容风险

不同终端对颜色、键盘事件、窗口尺寸支持不同。

应对：

- 使用 Crossterm；
- 控制颜色使用；
- 支持简化布局；
- 提供终端兼容说明。

## 2. 产品风险

### 2.1 普通用户配置门槛高

虚拟声卡配置对普通用户不友好。

应对：

- 首次启动向导；
- 配置检测；
- 一键测试；
- 图文教程；
- 后续开发 GUI。

### 2.2 TUI 学习成本

TUI 对不熟悉终端的用户存在门槛。

应对：

- 提供清晰快捷键提示；
- 支持命令面板；
- 保留帮助页面；
- 后续提供 GUI 客户端。

## 3. 后续演进方向

### 3.1 Tauri GUI 客户端

在 gamesound-core 稳定后，可开发 Tauri GUI：

- 图形化音效卡片；
- 拖拽导入；
- 设备配置向导；
- 快捷键可视化绑定；
- 音频波形与剪辑。

### 3.2 自研虚拟麦克风

长期可考虑自研虚拟麦克风：

- Windows 虚拟音频驱动；
- macOS CoreAudio / AudioDriverKit；
- 安装器；
- 签名与公证；
- 自动卸载。

该方向技术难度高，不建议早期实现。

### 3.3 音频剪辑

支持：

- 裁剪音频；
- 调整音量；
- 淡入淡出；
- 保存片段；
- 波形显示。

### 3.4 AI 语音生成

支持：

- 输入文本生成音效；
- 保存为音效；
- 绑定快捷键；
- 语音角色管理。

### 3.5 音效方案分享

支持：

- 导出音效包；
- 导入音效包；
- 云端同步；
- 本地分享。

## 4. 长期架构方向

推荐保持：

```text
gamesound-core
  ├── gamesound-tui
  ├── gamesound-desktop
  └── gamesound-daemon
```

其中 core 负责所有音频与运行时能力，TUI、GUI、daemon 都只是不同客户端。

## 5. 安全与隐私

由于软件会采集麦克风，应明确：

- 默认不上传任何音频；
- 麦克风数据仅用于本地混音；
- 日志不得记录用户音频内容；
- 录音功能如后续实现，必须显式提示用户；
- 导出音效包时应提示是否包含原始音频文件。
